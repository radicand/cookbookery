# Flutter iOS Dependency Upgrade Skill

## When to Use This Skill

- User wants to upgrade the iOS deployment target (`platform :ios` or `IPHONEOS_DEPLOYMENT_TARGET`)
- User wants to run `pod update` to upgrade CocoaPods pod versions
- User wants to upgrade fastlane gems in `Gemfile`
- iOS build fails after a Flutter SDK upgrade
- Xcode reports warnings about outdated pod versions or deprecated build settings
- App Store requires a higher minimum iOS version

## Prerequisite: Read the Project Files First

Always read these files before making changes:

```
read ios/Podfile
read ios/Gemfile
```

If `ios/Gemfile.lock` and `ios/Podfile.lock` exist, read them to understand the currently locked versions:
```
read ios/Podfile.lock
read ios/Gemfile.lock
```

Also check whether `android/Gemfile` and `android/fastlane/Pluginfile` differ from `ios/Gemfile` — this project maintains separate Gemfiles per platform.

## Key Files and Their Roles

| File | Controls |
|---|---|
| `ios/Podfile` | Platform target, pod dependencies, CocoaPods post-install hooks |
| `ios/Podfile.lock` | Exact resolved pod versions — **committed to git** for reproducibility |
| `ios/Gemfile` | Ruby gem dependencies for fastlane and CocoaPods |
| `ios/Gemfile.lock` | Exact resolved gem versions — **committed to git** |
| `ios/Flutter/Generated.xcconfig` | Auto-generated by `flutter pub get`; contains `FLUTTER_ROOT` — the Podfile needs this to locate the Flutter SDK |
| `ios/Runner/Info.plist` | App metadata; `MinimumOSVersion` here must align with the Podfile deployment target |
| `ios/Runner.xcworkspace` | The Xcode workspace to use — **always open this**, never `Runner.xcodeproj` directly |

## Upgrade Sequence (Always Follow This Order)

1. `flutter pub get` → 2. Gemfile update → 3. Podfile platform version → 4. Pod install/update → 5. Xcode build verification

Skipping step 1 causes `pod install` to fail because `ios/Flutter/Generated.xcconfig` must exist before CocoaPods can locate `FLUTTER_ROOT`.

---

## Step 1: Regenerate Flutter iOS Config First

**Always run before any pod or gem commands:**

```bash
cd <project_root>   # the directory containing pubspec.yaml
flutter pub get
```

This regenerates `ios/Flutter/Generated.xcconfig`. Without it, the Podfile's `flutter_root` function will raise an error.

---

## Step 2: Upgrade Gemfile (fastlane, CocoaPods)

**File:** `ios/Gemfile` (and separately `android/Gemfile` if present)

### Check current gem versions
```bash
cd ios
bundle exec fastlane --version
bundle exec pod --version
```

### Upgrade all gems to latest compatible versions
```bash
cd ios
bundle update          # upgrades all gems, rewrites Gemfile.lock
```

### Upgrade a specific gem only
```bash
bundle update fastlane
bundle update cocoapods
```

### Install gems from a fresh checkout (no update)
```bash
bundle install         # installs exact versions from Gemfile.lock
```

**After updating:**
- Review the `Gemfile.lock` diff before committing — confirm no unexpected version changes
- Always commit `Gemfile.lock` alongside `Gemfile`; it ensures CI and team members use identical gem versions

**CocoaPods via Bundler:** Always prefix pod commands with `bundle exec pod ...` when CocoaPods is managed via Gemfile. This ensures the Bundler-managed version is used, not a globally installed one:
```bash
bundle exec pod install --repo-update
bundle exec pod outdated
bundle exec pod update <PodName>
```

Latest fastlane releases: https://github.com/fastlane/fastlane/releases

## Step 3: Update Podfile Platform Version and Deployment Target

**File:** `ios/Podfile`

This project currently targets iOS 18.0. There are **two places** that must stay in sync:

### 1. Top-level platform declaration
```ruby
platform :ios, '18.0'   # ← change this version
```
This tells the CocoaPods resolver to reject pods that require a higher minimum iOS version than specified.

### 2. Post-install hook `IPHONEOS_DEPLOYMENT_TARGET`
```ruby
post_install do |installer|
  installer.pods_project.targets.each do |target|
    flutter_additional_ios_build_settings(target)
    target.build_configurations.each do |config|
      config.build_settings['IPHONEOS_DEPLOYMENT_TARGET'] = '18.0'  # ← change this to match
    end
  end
end
```
This overrides the per-pod Xcode build setting. Without it, individual pods may have a lower deployment target, causing Xcode linker warnings or symbol availability errors.

**Both values must be identical.** Update them together.

### When to change the deployment target
- When requiring a new iOS API only available in a higher version
- When a pod's minimum is higher than your current `platform` value — pod install will error
- When Apple raises the App Store minimum iOS requirement

**Apple App Store minimum iOS for new apps (2026):** iOS 16+  
Track requirements: https://developer.apple.com/news/upcoming-requirements/

## Step 4: Install or Update Pods

### Understand the difference

| Command | Effect |
|---|---|
| `bundle exec pod install` | Installs pods according to `Podfile.lock` (exact pinned versions). For new pods not yet in the lockfile, resolves the latest satisfying version. **Use for daily development.** |
| `bundle exec pod install --repo-update` | Same as `pod install`, but first updates the local CocoaPods master spec repo from the remote. Required when a new pod version isn't cached locally. |
| `bundle exec pod update` | Ignores `Podfile.lock` entirely and resolves the **latest version of every pod**. **Use only when intentionally upgrading all pods.** |
| `bundle exec pod update <PodName>` | Updates only the named pod (and its dependencies) to the latest compatible version. **Preferred for targeted upgrades.** |
| `bundle exec pod outdated` | Lists pods with newer versions available without modifying anything. |

### Recommended upgrade workflow

```bash
cd ios

# 1. Check what's outdated without changing anything
bundle exec pod outdated

# 2. Update specific pods (preferred — targeted and reviewable)
bundle exec pod update sentry_flutter
bundle exec pod update flutter_secure_storage

# 3. If upgrading all pods (e.g., after a major Flutter SDK upgrade)
bundle exec pod update

# 4. Or if you only need to install (not upgrade) after a Podfile change
bundle exec pod install --repo-update
```

After any pod update:
1. Review `Podfile.lock` diff: `git diff ios/Podfile.lock`
2. Confirm only expected pods changed versions
3. Commit `Podfile.lock`

## Step 5: Verify the iOS Build

```bash
# Preferred — Flutter's build wrapper (handles all Xcode project coordination):
flutter build ios --no-codesign           # full build, no signing required
flutter build ios --no-codesign --simulator  # simulator-only, faster

# For production archive (requires signing + certificates):
flutter build ipa

# Direct Xcode CLI (platform-level verification):
xcodebuild \
  -workspace ios/Runner.xcworkspace \
  -scheme Runner \
  -configuration Debug \
  -destination 'generic/platform=iOS Simulator' \
  build

# Check Xcode version (must meet Flutter's minimum Xcode requirement):
xcodebuild -version
```

**Always open `ios/Runner.xcworkspace`** in Xcode, never `Runner.xcodeproj`. The `.xcworkspace` includes all CocoaPods targets; `.xcodeproj` alone will be missing pod dependencies.

## Common Errors and Fixes

### `Generated.xcconfig must exist`
Pod install ran before `flutter pub get`. Run `flutter pub get` from the project root first. This regenerates `ios/Flutter/Generated.xcconfig`.

### `[!] CocoaPods could not find compatible versions for pod "X"`
Two possible causes:
1. The `platform :ios` version is lower than the pod's minimum — raise the platform version
2. Two pods require incompatible versions of a shared pod — use `bundle exec pod install --verbose` to identify the conflicting constraint chain

Fix by raising the `platform :ios` version or by explicitly pinning the problematic pod:
```ruby
# ios/Podfile
pod 'SomeSharedPod', '~> 1.5'   # pin to compatible version
```

### `The iOS Simulator SDK '...` is not installed`
Xcode does not have the simulator SDK for the target iOS version. Open Xcode → Platforms and download the required simulator.

### `IPHONEOS_DEPLOYMENT_TARGET is set to X.X, but your minimum deployment target is Y.Y`
The post-install hook's `IPHONEOS_DEPLOYMENT_TARGET` value doesn't match the `platform :ios` value, **or** a pod has a higher minimum than what is set. Align all three: `platform :ios`, the post-install hook, and raise if a pod demands it.

### `bundle exec pod: command not found`
Gems not installed. Run `bundle install` from the `ios/` directory.

### Xcode: `Module 'X' not found`
Pod is listed in `Podfile.lock` but its files are missing from `ios/Pods/`. Run `bundle exec pod install --repo-update` to restore.

### `ld: library not found for -lX` (linker error)
Pod compiled for the wrong architecture or SDK. Clean and rebuild:
```bash
flutter clean
flutter pub get
cd ios && bundle exec pod install --repo-update
flutter build ios --no-codesign
```

## Post-Upgrade Info.plist Alignment

After changing the deployment target, verify `ios/Runner/Info.plist`:
```xml
<key>MinimumOSVersion</key>
<string>18.0</string>   <!-- must match Podfile platform version -->
```

If it's out of sync, Xcode or App Store Connect validation will warn. Update it by editing `Info.plist` directly or via Xcode → Target → General → Deployment Info.

## Commit Checklist

After a successful upgrade, commit these files:

```bash
git add ios/Podfile             # if platform version changed
git add ios/Podfile.lock        # always commit after any pod change
git add ios/Gemfile             # if gem versions changed
git add ios/Gemfile.lock        # always commit after bundle update
git add ios/Runner/Info.plist   # if MinimumOSVersion changed
git commit -m "chore(ios): upgrade pods and deployment target to iOS X.X"
```

**Never commit** the `ios/Pods/` directory — it should be in `.gitignore`.
